# Exposure Gate

Status: active

Purpose
-------
This is the mandatory pre-exposure check for AllDone.

It exists to prevent the most common publication mistakes:

- overstating maturity
- exposing internal leverage or strategy
- making quality claims without scope
- using investor language that implies shipped proof
- publishing text that is technically correct but socially wrong

This gate combines the useful parts of UCAS exposure discipline with UFG unit-quality discipline.

## Gate outcome

Any outward-facing AllDone text should pass these checks before publication.

If a document fails one of the checks below, it should be revised before exposure.

## 1. Stage Reality Gate

Every sentence must match current reality.

Current allowed framing:

- concept
- research
- prototype
- IP-claim record
- funding-ready in narrative structure

Current disallowed framing:

- launch-ready
- production-ready
- pilot-proven
- already validated economics
- active pilot unless one actually exists

## 2. Internal Exposure Gate

Do not publish internal leverage, negotiation posture, or strategic manipulation language.

Allowed:

- what the system does
- what value it provides
- what it still needs

Not allowed:

- persuasion tactics
- internal negotiation framing
- strategic vulnerability notes
- recipient-specific leverage logic

## 3. Object-First Gate

Place the object on the table before the worldview.

For AllDone, that means:

- first say what the system is
- then say what value it provides
- then explain why it matters

Do not lead with manifesto language when a concrete system can be described plainly.

## 4. Promise Gate

Do not imply proof that does not exist.

Examples of unsafe phrasing:

- "works at scale"
- "production-ready"
- "validated business model"
- "pilot-ready" without live operating proof

Safe replacement pattern:

- state what exists now
- state what remains to be validated
- state the next proof step

## 5. Ask Gate

If the document is meant to attract help, the ask must match the stage.

Good current asks:

- build partners
- operator/advisor support
- pilot-access conversations
- pre-seed funding tied to proof milestones
- supplier and wash-path validation

Bad current asks:

- broad launch marketing
- franchise-style scale discussions
- commitments that assume a proven operating loop

## 6. Quality Claim Gate

Any quality or progress claim must have a real scope boundary.

Before exposure, ask:

1. What exactly was observed?
2. In what environment?
3. What is still unproven?
4. What would invalidate the claim?

Safe external pattern:

"The repo contains a structured concept, prototype logic, and pilot assumptions. Live operating proof still needs to be established."

## 7. Tone Gate

Reject text that feels:

- too corporate
- too manifesto-heavy
- too GPT-symmetric
- too negative-by-listing
- too clever for normal readers

Prefer:

- direct sentences
- normal human wording
- positive description of what is being built

## 8. UFG Unit Gate

At the sentence level, remove or repair:

- information loss
- hallucinated or distorted claims
- internal strategy exposure
- legal-risk phrasing
- audience mismatch
- tone drift
- emotional distortion
- binary negation patterns
- repetitive sentence structure
- negation-as-authority phrasing

## 9. Release Decision

For any public or investor-facing text, the final questions are:

1. Does it say what AllDone is before saying what AllDone thinks?
2. Does it claim only what is true now?
3. Does it hide what should remain internal?
4. Does it ask for the right kind of help?
5. Would a sharp outsider read it as credible rather than inflated?

If the answer to any of those is no, revise before exposure.
